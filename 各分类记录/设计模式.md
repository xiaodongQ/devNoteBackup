# 设计模式

## 1 面向对象基础

### 1.1 核心特性
封装、抽象、继承、多态

### 1.2 抽象类 vs 接口

| | 抽象类 | 接口 |
|---|---|---|
| 关系 | `is-a` | `has-a` |
| 目的 | 解决代码复用 | 解决耦合问题 |
| 内容 | 成员变量和方法 | 仅方法 |

- **抽象类**：是对成员变量和方法的抽象，是为了解决代码复用问题
- **接口**：仅仅是对方法的抽象，是为了隔离接口和具体的实现，提高代码的扩展性

参考：[08 | 理论五：接口vs抽象类的区别？](https://time.geekbang.org/column/article/165103)

### 1.3 关键设计思想
- 基于接口编程而非实现编程
- 多用组合，少用继承

参考：[10 | 理论七：为何说要多用组合少用继承？](https://time.geekbang.org/column/article/169593)

## 2 贫血模型与充血模型

### 2.1 贫血模型（Anemic Domain Model）

MVC 三层架构中的 M 表示 Model，V 表示 View，C 表示 Controller。将整个项目分为三层：展示层、逻辑层、数据层。

现代 Web/App 项目采用前后端分离，后端分为：
- **Repository 层**：负责数据访问
- **Service 层**：负责业务逻辑
- **Controller 层**：负责暴露接口

**什么是贫血模型？**
以 `UserBo` 为例，它是一个纯粹的数据结构，只包含数据，不包含任何业务逻辑。业务逻辑集中在 `UserService` 中。这种将数据与操作分离的模式，破坏了面向对象的封装特性，是一种典型的**面向过程**的编程风格。

参考：[11 | 实战一：基于贫血模型的MVC架构违背OOP吗？](https://time.geekbang.org/column/article/169600)

### 2.2 充血模型（Rich Domain Model）

数据和对应的业务逻辑被封装到同一个类中，满足面向对象的封装特性，是典型的**面向对象**编程风格。

### 2.3 领域驱动设计（DDD）

DDD 主要用来指导如何解耦业务系统，划分业务模块，定义业务领域模型及其交互。微服务加速了 DDD 的盛行。

**DDD 与 MVC 的关系：**
基于充血模型的 DDD 开发模式也按照 MVC 三层架构分层，区别主要在 Service 层：
- 传统模式：重 Service 轻 BO
- DDD 模式：轻 Service 重 Domain

**适用场景：**
- 贫血模型：业务比较简单的系统
- 充血模型（DDD）：业务复杂的系统（如金融系统的利息计算、还款模型等）

## 3 设计原则

### 3.1 SOLID 原则

#### 3.1.1 单一职责原则（SRP）

> 一个类或者模块只负责完成一个职责

- 不要设计大而全的类，要设计粒度小、功能单一的类
- 目的：实现代码高内聚、低复用、可读、可维护
- 不同应用场景对"职责是否单一"的判定可能不同

参考：[15 | 理论一：如何判定某个类的职责是否够"单一"？](https://time.geekbang.org/column/article/171771)

#### 3.1.2 开闭原则（OCP）

> 软件实体应该"对扩展开放、对修改关闭"

添加新功能应该是扩展代码（新增模块、类、方法），而非修改已有代码。

参考：[16 | 理论二：如何做到"对扩展开放、修改关闭"？](https://time.geekbang.org/column/article/176075)

#### 3.1.3 里式替换原则（LSP）

> 子类对象能够替换程序中父类对象出现的任何地方，保证逻辑行为不变及正确性不被破坏

- 多态是语法，里式替换是设计原则
- 核心：design by contract，按照协议来设计
- 父类定义函数的"约定"，子类可以改变内部实现，但不能改变原有"约定"

参考：[17 | 理论三：里式替换（LSP）跟多态有何区别？](https://time.geekbang.org/column/article/177110)

#### 3.1.4 接口隔离原则（ISP）

> 客户端不应该强迫依赖它不需要的接口

**应用场景：**
- 将特权操作单独定义接口，实现类可选择实现
- 微服务/类库接口设计中，部分接口只被部分调用者使用

**示例：** 支持 Redis 和 Kafka 配置热更新，但不希望 MySQL 热更新
```
interface Updater { void update(); }
RedisConfig、KafkaConfig 实现 Updater → 热更新
MysqlConfig 不实现 Updater → 不支持热更新
```

参考：[18 | 理论四：接口隔离原则有哪三种应用？](https://time.geekbang.org/column/article/177442)

#### 3.1.5 依赖反转原则（DIP）

**控制反转（IOC）**
框架提供可扩展的代码骨架，程序员添加业务相关代码来驱动程序流程。流程控制权从程序员"反转"到了框架。

**依赖注入（DI）**
不通过 new() 在类内部创建依赖对象，而是在外部创建好后通过构造函数/参数传递进来。

**依赖反转原则**
高层模块不要依赖低层模块，两者都应该依赖抽象；抽象不要依赖具体实现细节，具体实现细节依赖抽象。

参考：[19 | 理论五：控制反转、依赖反转、依赖注入的区别？](https://time.geekbang.org/column/article/177444)

---

### 3.2 KISS 原则（Keep It Simple）

> 尽量保持简单

**不是代码行数越少就越简单**，比如正则表达式代码少但本身复杂。

**如何写出满足 KISS 的代码：**
1. 不要使用同事可能不懂的技术
2. 不要重复造轮子，善于使用已有工具类库
3. 不要过度优化（位运算、复杂的条件语句等牺牲可读性）

参考：[20 | 理论六：KISS、YAGNI原则看似简单，却经常被用错](https://time.geekbang.org/column/article/177448)

---

### 3.3 YAGNI 原则（You Ain't Gonna Need It）

> 不要设计当前用不到的功能；不要编写当前用不到的代码

不要做过度设计，但可以预留扩展点。

- KISS 原则讲的是"如何做"（尽量保持简单）
- YAGNI 原则说的是"要不要做"（当前不需要的就不要做）

---

### 3.4 DRY 原则（Don't Repeat Yourself）

> 不要重复自己，不要写重复的代码

**三种代码重复的情况：**
1. 实现逻辑重复，但功能语义不重复 → 不违反 DRY
2. 实现逻辑不重复，但功能语义重复 → 违反 DRY
3. 代码执行重复 → 违反 DRY

**提高代码可复用性的方法：**
1. 减少代码耦合
2. 满足单一职责原则
3. 模块化业务与非业务逻辑分离
4. 通用代码下沉
5. 继承、多态、抽象、封装
6. 应用模板等设计模式

**意识比技术更重要：** 我们可以不写可复用的代码，但一定不能写重复的代码。

参考：[21 | 理论七：重复的代码就一定违背DRY吗？](https://time.geekbang.org/column/article/179607)

---

### 3.5 迪米特法则（LOD）

也称为"最小知识原则"：每个模块只应该了解那些与它关系密切的模块有限的知识。

核心：**不该有直接依赖关系的类之间不要有依赖；有依赖关系的类之间，尽量只依赖必要的接口**

目的：实现高内聚、松耦合

参考：[22 | 理论八：如何用迪米特法则实现"高内聚、松耦合"？](https://time.geekbang.org/column/article/179615)

## 4 创建型模式

创建型模式主要解决对象的创建问题，封装复杂的创建过程，解耦对象的创建代码和使用代码。

包括：**单例模式**、**工厂模式**、**建造者模式**、**原型模式**

---

### 4.1 单例模式（Singleton）

一个类只允许创建一个对象（实例）。

**为什么要使用单例？**

1. **处理资源访问冲突**
   - 将对象级别锁换成类级别锁，所有线程共享同一把锁
   - 其他方案：分布式锁、并发队列

2. **表示全局唯一类**
   - 配置信息类
   - 唯一递增 ID 号码生成器

**实现方式：**

| 方式 | 延迟加载 | 线程安全 | 并发性能 |
|---|---|---|---|
| 饿汉式 | 否 | 是 | 高 |
| 懒汉式 | 是 | 是（加锁） | 低 |
| 双重检测 | 是 | 是 | 高 |
| 静态内部类 | 是 | 是 | 高 |
| 枚举（Java） | 是 | 是 | 高 |

**饿汉式：**
类加载时静态实例已创建初始化，线程安全，但不支持延迟加载。

适用场景：实例占用资源多或初始化耗时长，提前初始化可避免运行时的性能问题。

**懒汉式：**
支持延迟加载，但 `getInstance()` 加锁导致并发度低、性能瓶颈。

**双重检测：**
既支持延迟加载、又支持高并发。instance 创建后不会再加锁。

**静态内部类：**
既保证线程安全，又做到延迟加载。

**枚举（Java）：**
最简洁的实现方式。

**单例存在的问题：**

1. **对 OOP 特性支持不友好** - 违背基于接口而非实现的设计原则
2. **隐藏类之间的依赖关系** - 调用关系隐蔽
3. **对代码扩展性不友好** - 无法适应需要创建多个实例的场景（如数据库连接池）
4. **对代码可测试性不友好** - 难以 mock 替换
5. **不支持有参数的构造函数** - 解决思路：将参数放到另一个全局变量中（推荐）

**替代解决方案：**
- 静态方法：无法解决之前的问题
- 依赖注入：可以解决隐藏依赖关系的问题，但无法解决其他问题

参考：[41-43 | 单例模式](https://time.geekbang.org/column/article/194035)

---

### 4.2 工厂模式（Factory）

**分类：** 简单工厂、工厂方法、抽象工厂

#### 4.2.1 简单工厂

将创建不同情况的类剥离到独立类中，只负责对象的创建。工厂类名通常以 `Factory` 结尾，创建方法一般为 `create` 开头。

如需频繁创建实例，可将不同情况的类事先创建好缓存起来（利用 static 的 map 变量）。

适用场景：创建逻辑复杂、有 if-else 分支判断、需要根据类型动态创建对象。

#### 4.2.2 工厂方法

利用多态消除 if 分支判断，比简单工厂更符合开闭原则。

可以通过工厂的工厂（简单工厂）来避免使用时判断返回不同类型的工厂类。

#### 4.2.3 抽象工厂

当类有两种分类方式时（如按配置文件格式 + 按解析对象类型），抽象工厂可以让一个工厂负责创建多个不同类型的对象，有效减少工厂类个数。

**工厂模式的作用：**
1. 封装变化：创建逻辑变更对调用者透明
2. 代码复用：创建代码抽离到独立工厂类后可复用
3. 隔离复杂性：调用者无需了解如何创建对象
4. 控制复杂度：职责更单一，代码更简洁

**选择建议：**
- 创建逻辑简单 → 直接 `new`，不需要工厂模式
- 创建逻辑复杂 + 只需创建一种对象 → 简单工厂
- 创建逻辑复杂 + 需要创建多种相关对象 → 工厂方法

参考：[44-45 | 工厂模式](https://time.geekbang.org/column/article/197254)

---

### 4.3 建造者模式（Builder）

**适用场景（满足任一即可）：**
1. 必填属性很多，构造函数参数列表过长
2. 属性之间有依赖关系或约束条件
3. 希望创建不可变对象（不暴露 set 方法）

**用法：**
1. 将校验逻辑放置到 Builder 类中
2. 在 `build()` 方法中进行集中校验，校验通过后才创建对象
3. 构造函数改为 `private`，只能通过建造者创建
4. 不提供任何 set 方法，创建出来的对象就是不可变对象

Builder 类可以设置为内部静态类，也可以设计成独立的类。

**作用：** 避免对象存在无效状态，一次性初始化所有成员变量。

**与工厂模式的区别：**
- 工厂模式：创建不同但相关类型的对象（继承同一父类/接口的子类），由参数决定创建哪种
- 建造者模式：创建一种类型的复杂对象，通过可选参数定制化创建不同对象

参考：[46 | 建造者模式](https://time.geekbang.org/column/article/199674)

---

### 4.4 原型模式（Prototype）

**原理：** 如果对象的创建成本比较大，而同一个类的不同对象之间差别不大，可以利用对已有对象进行复制的方式来创建新对象。

**应用场景：** 数据需要经过复杂计算（排序、哈希）才能得到，或需要从 RPC、网络、数据库、文件系统慢速 IO 中读取。

**实现：深拷贝 vs 浅拷贝**

浅拷贝的问题：散列表中存储的是对象引用，拷贝后 newKeywords 和 currentKeywords 指向相同对象，导致数据版本不一致。

**深拷贝的两种实现方法：**
1. 递归拷贝对象及其引用对象，直到只包含基本数据类型
2. 先序列化再反序列化

**优化策略：** 先用浅拷贝创建 newKeywords，对于需要更新的对象再用深拷贝创建新对象替换。

参考：[47 | 原型模式](https://time.geekbang.org/column/article/200786)

## 5 结构型模式

结构型模式总结了一些类或对象组合在一起的经典结构，可以解决特定应用场景的问题。

包括：**代理模式**、**桥接模式**、**装饰器模式**、**适配器模式**、**门面模式**、**组合模式**、**享元模式**

---

### 5.1 代理模式（Proxy）

在不改变原始类的情况下，通过引入代理类来给原始类附加功能。

**实现方式：**
1. **相同接口**（优先）：代理类和原始类实现相同接口，参照基于接口而非实现编程
2. **继承**：如果原始类没有接口且无法修改（如第三方类库），可以让代理类继承原始类

**动态代理：**
不事先为每个原始类编写代理类，而是在运行时候动态创建代理类，替换掉原始类。Java 提供动态代理语法，底层依赖反射。

**应用场景：**
1. **业务系统的非功能性需求**：监控、统计、鉴权、限流、事务、幂等、日志
2. **RPC 框架**：通过远程代理隐藏网络通信、数据编解码细节
3. **缓存**：AOP 切面拦截请求，从缓存中获取数据直接返回

参考：[48 | 代理模式](https://time.geekbang.org/column/article/201823)

---

### 5.2 桥接模式（Bridge）

**定义：** 将抽象和实现解耦，让它们可以独立变化

**另一种理解：** 一个类存在两个（或多个）独立变化的维度，通过组合方式让这两个维度可以独立扩展。类似于"组合优于继承"原则，避免继承层次的指数级爆炸。

**经典应用：JDBC 驱动**
- JDBC 相当于"抽象"
- 具体的 Driver（如 com.mysql.jdbc.Driver）相当于"实现"
- JDBC 和 Driver 独立开发，通过对象之间的组合关系组装在一起

**应用示例：消息推送系统**

场景：Notification 类支持多种渠道（邮件、短信、微信、电话）和多种紧急程度（SEVERE、URGENCY、NORMAL、TRIVIAL）。

方案：
1. 将不同渠道的发送逻辑剥离为独立的 MsgSender 接口
2. Notification 作为抽象，包含 MsgSender 实现
3. 根据紧急程度继承 Notification 抽象类

```
MsgSender（接口）
  ├── EmailMsgSender
  ├── SmsMsgSender
  └── ...

Notification（抽象类）
  ├── contains: MsgSender
  ├── SevereNotification
  ├── UrgencyNotification
  └── ...
```

两个维度独立扩展，互不影响。

参考：[49 | 桥接模式](https://time.geekbang.org/column/article/202786)

---

### 5.3 装饰器模式（Decorator）

**特点：**
1. 装饰器类和原始类继承同样的父类，可以对原始类"嵌套"多个装饰器
2. 装饰器类是对功能的增强

**区别：** 代理模式附加的是跟原始类无关的功能，装饰器模式附加的是跟原始类相关的增强功能。

**作用：** 解决继承关系过于复杂的问题，通过组合来替代继承。主要给原始类添加增强功能。

**注意：** 可以对原始类嵌套使用多个装饰器，装饰器类和原始类需要继承相同的抽象类或接口。

参考：[50 | 装饰器模式](https://time.geekbang.org/column/article/204845)

---

### 5.4 适配器模式（Adapter）

**作用：** 将不兼容的接口转换为可兼容的接口，让原本不能一起工作的类可以一起工作。

**两种实现方式：**

| | 类适配器 | 对象适配器 |
|---|---|---|
| 实现 | 继承 | 组合 |
| 适用 | Adaptee 和 ITarget 接口大部分相同 | Adaptee 和 ITarget 接口大部分不同 |

选择标准：Adaptee 接口个数，以及两者契合程度。

**应用场景：**
1. **封装有缺陷的接口设计** - 对外部系统接口进行二次封装
2. **统一多个类的接口设计** - 将多个外部系统的接口适配为统一接口
3. **替换依赖的外部系统** - 减少代码改动
4. **兼容老版本接口** - 旧接口标记为 deprecated，内部委托新接口实现
5. **适配不同格式的数据** - 不同征信系统数据格式统一

**四种模式对比：**

| 模式 | 目的 | 与原始类接口 |
|---|---|---|
| 代理模式 | 控制访问 | 相同 |
| 桥接模式 | 接口和实现分离 | 相同 |
| 装饰器模式 | 功能增强 | 相同 |
| 适配器模式 | 接口兼容 | **不同** |

都是 Wrapper 模式，通过二次封装原始类来实现。

参考：[51 | 适配器模式](https://time.geekbang.org/column/article/205912)

## 6 C++ 单例模式

```cpp
// C++11 下的实现（Meyers' Singleton）
// 函数内的 local static 对象，线程安全（C++0x之后）
// C++0x之前仍需加锁
class Singleton {
private:
    Singleton() { };
    ~Singleton() { };
    Singleton(const Singleton&);
    Singleton& operator=(const Singleton&);
public:
    static Singleton& getInstance() {
        static Singleton instance;
        return instance;
    }
};
```
